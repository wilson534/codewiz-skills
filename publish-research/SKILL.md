# Publish Research

将一次技术调研的中间产物（多份散落的草稿文档）整理成一份对外可发的单文档，完成语言打磨，推送到指定仓库。

## 触发时机

用户说「帮我把调研整理成文档」「要发出去了，帮我润色」「push 到 GitLab/GitHub」时加载此 skill。

## 工作流程

### 第一步：摸清现状

先做这三件事，不要急着动文件：

1. 列出工作目录下所有 `.md` 文件，确认哪些是草稿、哪些是已有的对外文档
2. 明确对外文档的目标读者：组内同学？外部开发者？
3. 确认发布目标：GitLab 仓库路径、GitHub 仓库、还是两者都要

### 第二步：结构整合

将多份草稿合并成一份文档，结构参考：

```
## 它是什么，解决什么问题      ← 一段话说清背景，不超过 150 字
## 和现有方案哪里不同          ← 对比表格，列维度而非罗列功能
## 核心机制详解                ← 每个机制：是什么 + 为什么这样设计 + 有什么限制
## 设计空白                    ← 明确说出框架没有做的事，不要回避
## 个人判断                    ← 适用场景、不适用场景、值得借鉴的地方
```

不要机械套用这个结构——根据调研内容调整，但每个节必须能回答「读者看完这节知道了什么」。

### 第三步：语言打磨

逐段检查，重点看这几类问题：

**技术术语没有解释**
- 改前：「原子写（先写 `.tmp` 再 rename）」
- 改后：「写入时先落到临时文件再 rename，避免写一半时进程崩溃导致文件损坏」
- 规则：技术名词后面跟一句「为什么这样做」

**结论没有原因**
- 改前：「接入方需要自己想清楚每类知识应该用什么策略」
- 改后：「灵活性提高了，但每类知识该用什么策略，接入方必须自己想清楚，框架不会替你决定」
- 规则：判断句后面跟「因为……」或「代价是……」

**现象没有说清影响**
- 改前：「三个月前的 entry 和昨天写的对 LLM 没有任何区别」
- 改后：「LLM 看到的内容里没有任何时间信息，三个月前写的和昨天写的没有区别」
- 规则：先说 LLM 视角的结果，再说技术原因

**晦涩词换直白词**
- 「正交」→「独立」
- 「prompt 层面的分类指令」→「给 Claude 的写入提示，框架不做校验」
- 「最终写入的一致性模型」→ 解释清楚「丢的只是 git 历史，不是内容」

**没有价值的实现细节**
- 「实现：三种策略全部在 `buildRootPrompt()` 里处理，约 45 行代码」→ 直接删掉
- 规则：对不看源码的读者没有价值的行，删

**不造数字，不加评估腔**
- 不写「显著提升」「大幅优化」「ROI 极高」
- 不写「阅读时长：15分钟」
- 数字必须来自源码或实测

### 第四步：核实每一条结论的证据

**每个结论都必须能指回源头，不能只有判断没有依据。**

证据有两种形式：

**源码位置**（最强）
- 格式：`文件名:行号`，例如 `WorkspaceManager.ts:140-185`
- 做法：写下结论后，立即打开源码确认行号；行号会随版本变化，注明分析时的 commit 或日期
- 反例：「WM 注入时不读 `updatedAt`」→ 没有证据
- 正例：「`buildRootPrompt()`（`WorkspaceManager.ts:140-185`）注入时完全不读 `updatedAt`——字段在 `save()` 时写入 `Date.now()`，但这个值从未被消费」

**引用原始文本**（次之）
- 引用 system prompt、代码注释、官方文档原文，用代码块包裹
- 不要转述，转述会引入失真

**核实清单——发布前逐条过**：
- [ ] 对比表格里的每一行，都能在源码里找到对应位置
- [ ] 「没有实现」的结论，已确认是搜过整个源码而非猜测（用 `grep` 或全局搜索验证）
- [ ] 引用的行号已对照当前版本确认无误
- [ ] 文档里注明了分析基于哪个版本 / commit / 日期

**「没有」比「有」更难证明**——说「WM 没有维护机制」需要搜过所有文件确认，不能因为没看到就下结论。

### 第五步：发现并点出设计空白

调研中最有价值的发现往往是「写了但没用」「声明了但没有消费者」这类反差：

- 检查每个字段/接口：有没有写入但注入时不读的？（如 `updatedAt`）
- 检查每个阶段：有没有完全空白的？（如维护阶段）
- 检查每个配置项：有没有配置了但框架完全不管的？

这类发现要在文档里明确点出，不要含糊带过。

### 第五步：发布

**GitLab（公司内网，SSH）**

```bash
# SSH key 路径按实际配置
GIT_SSH_COMMAND="ssh -i ~/.ssh/id_ed25519_xhs" git add <文件>
git -c user.name="<用户名>" -c user.email="<邮箱>" commit -m "<message>"
GIT_SSH_COMMAND="ssh -i ~/.ssh/id_ed25519_xhs" git push origin main
```

**GitHub（个人，token）**

如果仓库已存在：
```bash
git remote add github "https://<token>@github.com/<user>/<repo>.git"
git push github main
```

如果仓库不存在，先用 API 建：
```bash
curl -s -X POST https://api.github.com/user/repos \
  -H "Authorization: token <token>" \
  -H "Content-Type: application/json" \
  -d '{"name":"<repo>","private":true}'
```

**注意**：token 不要写进文件，只在命令行临时使用。

## 语言标准（判断一句话好不好的四个问题）

1. **读者不看源码，能理解这句话吗？** 不能 → 加解释
2. **这句话说了结论，但说了原因吗？** 没有 → 补原因
3. **这句话对读者有什么用？** 没用 → 删掉
4. **这个结论有没有源码或原文做依据？** 没有 → 补证据或降级为「推测」

## 常见陷阱

- **不要把草稿里的所有内容都保留**：草稿是给自己看的，对外文档要做减法
- **对比表格不是越多越好**：只对比读者真正需要做决策的维度
- **设计空白不要轻描淡写**：「WM 没有维护阶段」比「WM 的维护能力相对有限」更准确
- **不要在文档里替读者做判断**：「推荐使用 WM」不如「如果需要多团队共享记忆，WM 的 GitProvider 是 CC 没有的能力」
