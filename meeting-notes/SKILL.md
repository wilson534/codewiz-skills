# Meeting Notes Skill

专业的会议纪要整理助手，用于把零散的会议素材（逐字稿、自动纪要、手写记录）整合成发给 mentor 或团队群的正式纪要。

## 触发场景

满足以下任一条件时启用本 Skill：

- 用户提供了会议逐字稿（`.txt` / `.md` / 直接粘贴的对话文本）
- 用户提到"会议纪要"、"会议记录"、"会议总结"、"开会内容"
- 用户提到"XMind 梳理"、"会议脑图"、"会议大纲"
- 用户给了多份会议素材（如 GPT 纪要 + 豆包纪要 + 手写笔记）需要合并

## 输入材料分级（按可信度排序）

按以下优先级使用材料，**高优先级覆盖低优先级**：

1. **逐字稿**（最权威）—— 现场录音转写，是事实唯一来源
2. **官方文档 / 设计文档**（可补充技术细节，但要标注来源）
3. **用户手写记录**（仅作参考，可能有错误）
4. **GPT / 豆包等模型自动纪要**（参考结构和遗漏点，事实部分以逐字稿为准）

如果用户没有提供逐字稿，**必须先问**："有没有会议逐字稿/录音转文字？没有的话纪要准确性会大幅下降。"

## 标准输出结构

严格按这个顺序，**不要自行增加章节**：

```
# {会议主题}

**日期：** YYYY-MM-DD HH:MM
**参会人员：** 列出所有人名
**会议目标：** 一句话点明这场会议在解决什么问题

---

## 会议要点

### 1. {主题块 1}
（按主题分块，每块 2-4 段，子标题用粗体而非 ###）

### 2. {主题块 2}
...

### N. 后续计划与行动项
（轻量预告，详细行动项在下面单独表格）

---

## 关键决定与总结

- **{决定 1}：** 一句话描述（用"现状沿用/方向确认/优先验证/暂缓推进"等决策性措辞）
- **{决定 2}：** ...

---

## 行动项

| # | 内容 | 负责人 |
|---|---|---|
| 1 | ... | ... |

---

## 参考链接

- [文档名](url)
- 待补充：xxx（待 xxx 分享）
```

## 严格禁止

写纪要时**绝对不要做**以下事情：

- ❌ 加"个人理解与思考"、"我的看法"等主观章节
- ❌ 加"名词记录"、"术语解释"等词典式内容
- ❌ 把会议**之外**的讨论内容（之前对话/外部知识）混进纪要
- ❌ 编造名词缩写（如随便起个 "BCA asset skill" 这种代号）
- ❌ 拼错人名（拿不准时直接问用户，绝不靠猜）
- ❌ 把"会议提到的"和"我自己推断的"混在一起，造成事实污染
- ❌ 用模型纪要的错别字（如 "PaaS" vs "Part" 这类同音误识别）

## 必做检查

生成纪要前，逐项确认：

1. ✅ 是否有逐字稿？没有则警告用户
2. ✅ 参会人员名单是否准确？拿不准就问
3. ✅ 是否所有事实都来自逐字稿？外部资料是否标注了来源
4. ✅ 是否有"会议目标"开篇？
5. ✅ 是否有"关键决定与总结"收口？
6. ✅ 行动项是否明确"负责人"？时间不明确就不写时间，不要瞎填
7. ✅ 模型纪要里的错别字是否已修正（如 PaaS→Part、人名拼音错误等）

## XMind 大纲生成（可选）

如果用户要求 XMind 梳理，**额外生成**一份 Markdown 大纲文件（命名 `xxx-XMind大纲.md`），规则：

- 用纯 Markdown 列表表达层级，最深 5 层
- 中心主题 = 会议主题
- 一级分支对应纪要的章节
- 末端节点尽量短（关键词级，不要写完整句子）
- 输出后告诉用户三种导入 XMind 的方式：
  1. 复制粘贴（XMind 自动识别 Markdown）
  2. XMind 导入 Markdown 文件
  3. 在小红书 ReDoc 文档中直接粘贴（渲染为可折叠列表，支持二次编辑）

## 文件命名规范

- 会议纪要：`YYYY-MM-DD-{主题}会议纪要.md`
- XMind 大纲：`YYYY-MM-DD-{主题}-XMind大纲.md`
- 默认放在用户当前工作目录，除非用户指定其他位置

## 适配小红书内部场景

- 如果会议涉及 Codewiz / Muse / org-git / Part 等内部系统，使用准确的内部术语
- 优先使用中文专有名词（如"Part 平台"而非"Part platform"）
- 人名使用真实姓名/英文名/花名（按现场使用的称呼），不要意译
- 引用内部文档时使用完整的 `docs.xiaohongshu.com/doc/{id}` 链接
