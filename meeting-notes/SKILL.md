# Meeting Notes Skill

专业的会议纪要整理助手，用于把零散的会议素材（逐字稿、自动纪要、手写记录）整合成发给 mentor 或团队群的正式纪要。

## 触发场景

满足以下任一条件时启用本 Skill：

- 用户提供了会议逐字稿（`.txt` / `.md` / 直接粘贴的对话文本）
- 用户提到"会议纪要"、"会议记录"、"会议总结"、"开会内容"
- 用户提到"XMind 梳理"、"会议脑图"、"会议大纲"
- 用户给了多份会议素材（如 GPT 纪要 + 豆包纪要 + 手写笔记）需要合并

## 输入材料分级（按可信度排序）

按以下优先级使用材料，**高优先级覆盖低优先级**：

1. **逐字稿**（最权威）—— 现场录音转写，是事实唯一来源
2. **官方文档 / 设计文档**（可补充技术细节，但要标注来源）
3. **用户手写记录**（仅作参考，可能有错误）
4. **GPT / 豆包等模型自动纪要**（参考结构和遗漏点，事实部分以逐字稿为准）

如果用户没有提供逐字稿，**必须先问**："有没有会议逐字稿/录音转文字？没有的话纪要准确性会大幅下降。"

## 标准输出结构

严格按这个顺序，**不要自行增加章节**：

```
# {会议主题}

**日期：** YYYY-MM-DD HH:MM
**参会人员：** 列出所有人名
**会议目标：** 一句话点明这场会议在解决什么问题

---

## 会议要点

### 1. {主题块 1}
（按主题分块，每块 2-4 段，子标题用粗体而非 ###）

### 2. {主题块 2}
...

### N. 后续计划与行动项
（轻量预告，详细行动项在下面单独表格）

---

## 关键决定与总结

- **{决定 1}：** 一句话描述（用"现状沿用/方向确认/优先验证/暂缓推进"等决策性措辞）
- **{决定 2}：** ...

---

## 行动项

| # | 内容 | 负责人 |
|---|---|---|
| 1 | ... | ... |

---

## 参考链接

- [文档名](url)
- 待补充：xxx（待 xxx 分享）
```

## 严格禁止

写纪要时**绝对不要做**以下事情：

- ❌ 加"个人理解与思考"、"我的看法"等主观章节
- ❌ 加"名词记录"、"术语解释"等词典式内容
- ❌ 把会议**之外**的讨论内容（之前对话/外部知识）混进纪要
- ❌ 编造名词缩写（如随便起个 "BCA asset skill" 这种代号）
- ❌ 拼错人名（拿不准时直接问用户，绝不靠猜）
- ❌ 把"会议提到的"和"我自己推断的"混在一起，造成事实污染
- ❌ 用模型纪要的错别字（如 "PaaS" vs "Part" 这类同音误识别）
- ❌ 把语音转写的可疑英文术语 / 中文专名**直接照搬**进纪要（如 XQ→Explore、追命→Dreaming 这类误识别）
- ❌ 凭议题在会议中的**出场顺序 / 讨论时长**猜一期 / 二期归属
- ❌ 把模糊的发言拼成 action item（who / what / when **三要素任一缺失**就别写，宁缺勿凑）
- ❌ 在纪要正文用"X 说"、"X 强调"、"X 担心"等**归属语气**——除非原文有明确署名/分配，否则一律改用"会议讨论 / 达成共识 / 反对意见 / 决定"等中性陈述，避免误归属
- ❌ 把"待确认事项"留在终稿里——这是起草阶段的内部产物，发布前必须删除（疑问要么会前问完，要么会后跟提供者一对一确认）

## 必做检查

生成纪要前，逐项确认：

1. ✅ 是否有逐字稿？没有则警告用户
2. ✅ **多份素材是否同一场会议**？（看时间标签是否各自从 0 起、议程是否衔接）—— 同场合并，不同场分开
3. ✅ 参会人员名单是否准确？拿不准就问（**先问再写，不要用占位**）
4. ✅ 是否所有事实都来自逐字稿？外部资料是否标注了来源
5. ✅ **可疑的术语 / 专名是否对照过参考资料校准**？（如 XQ→Explore、追命→Dreaming）
6. ✅ **一期 / 二期归属是否有明确表态依据**？（不能凭出场顺序或讨论时长猜）
7. ✅ 是否有"会议目标"开篇？
8. ✅ 是否有"关键决定与总结"收口？
9. ✅ 行动项是否明确"负责人"？时间不明确就不写时间，不要瞎填
10. ✅ 行动项的动词 / 宾语 / 修饰语是否回原文核对过？（避免把"旧实现"的形容词误标成"新动作"目标）
11. ✅ 模型纪要里的错别字是否已修正（如 PaaS→Part、人名拼音错误等）
12. ✅ **可疑术语校准后是否全文 grep 确认替换干净**？（如把 "start-up" 校准为 "startup 业务知识上翻"，必须搜全文确认所有"start-up"都已统一替换，不能只改局部——局部漏改导致同一份纪要前后矛盾）
13. ✅ **action items 是否每条都有明确 who + what + when**？任一不全的删除，不要拼凑

## 常见陷阱（来自实战）

以下是真实使用中踩过的坑，按踩坑频率与影响排序。每条都附判断信号和处理方式。

### 1. 不要直接照搬可疑的语音转写

逐字稿中遇到看起来不合理的英文术语 / 中文专名，**先假设是语音转写错误**，对照参考资料 / 内部方案文档反向还原。

真实案例：
- "XQ Pro / XQ SQL" 实际是 **Explore / Explore Skill**
- "Python（下面）" 在 knowledge 目录上下文中，实际是 **experience / patterns** 这类目录名
- "追命" 实际是 **Dreaming**
- "客户 case" 实际是 **客户端 case**
- "玖月" 实际是 **久夜**（人名）

判断信号：术语在上下文里"读着别扭"、专名突然出现且不解释、英文单词拼写不像合规产品名、和已知方案文档的术语对不上。

处理方式：100% 不确定时标"**待确认 / 疑似 XX**"，绝不直接照搬。直接照搬比标待确认更糟。

### 2. 不要凭出场顺序判断议题归属（一期 / 二期 / 后续）

议题的时间归属（一期 / 二期 / 后续）**必须基于会议明确表态**（"先不做"、"下一期"、"一期就做"）**+ 对照外部规划文档**（OKR / 双月计划 / 路线图）。

**禁止使用**的判断方式：
- ❌ "议题在会议后半段提到 → 优先级低 → 归二期"
- ❌ "讨论时间短 → 不重要 → 归后续"
- ❌ "话题被一句带过 → 不在范围内"

正确的判断方式：
- ✅ 会议中 mentor / leader 明确说"这个一期不做"
- ✅ 议题对应外部 OKR / 双月规划文档中的明确条目
- ✅ 议题与已确定一期事项有明确依赖

### 3. 参考资料的"反向校准"用法

参考资料 / 内部方案文档对纪要有两种用途，要严格区分：

- ✅ **反向校准**：用方案文档纠正逐字稿的语音误识别（如 XQ→Explore、追命→Dreaming），还原会议中实际讨论的术语
- ❌ **添加内容**：把方案文档中讨论了但会议没提到的内容写进纪要

区分原则：**参考资料只能改名词的写法，不能给纪要添新事实。**

### 4. 多份素材的关系判断必须前置

拿到多份素材（多份逐字稿、多份录音转写）时，**第一步是判断它们的关系**：

- 是同一场会议的不同片段？→ 合并成一份纪要
- 是不同会议？→ 分别整理
- 是同一会议的转写 + 自动纪要？→ 以逐字稿为准

判断信号：
- 时间标签是否各自从 0 起（暗示是分段录音）
- 议题是否在两份之间衔接（"接着说第 4 点"出现在另一份）
- 主题连贯性

**不确定时直接问用户。** 不要先做两份再合并——白做。

### 5. 基础元数据"先问后做"

以下信息错了会导致全文返工，**应在动笔前确认**：

- 参会人真实姓名（不要用占位 / 花名猜测）
- 会议日期（如果不在文件名 / 元数据里）
- 关键议题的一期 / 二期归属（如果会议中有明确表态）

剩下的事实细节（讨论内容、决策措辞），可以"边写边标待确认"。

### 6. 行动项描述要逐字回原文核对

行动项的动词 / 宾语 / 修饰语归属容易混淆。最常见的错误：**描述"旧实现"的形容词被误标成"新动作"的目标**。

例：会议原话"把 plan 做成 CC 模式下，我之前做了一个 callback 类型的"
- ❌ 错误描述："把 plan 改造为 Claude Code 插件模式（callback 类型）" —— "callback 类型"是描述旧实现，不是新动作的目标
- ✅ 正确描述："把已有 Demo（mentor 之前 callback 类型的实现）迁移到 Claude Code 插件形式"

写完每条行动项，**回原文核对一次动词主语和修饰语归属**。

### 7. 议题间的合并 / 关联关系要识别

会议中如果某议题明确说"这是 X 的核心切入点"或"和 Y 是同一件事"，应当**合并到一条要点**而非独立列出。

例：会议中讨论"是否借鉴 CC 的 SUB Agent 机制"时，明确答："核心点是在 SPEC 结束的时候"——这就是 hook 蒸馏的同一切入点，不应作为独立议题列出。

判断信号：议题 A 的回答里出现了议题 B 的关键词 / 时机 / 触发点。

### 8. 内部术语的语义优先于字面意思

公司内部术语经常和字面英文/中文含义不同，必须用术语字典的语义而不是字面解释。

真实案例：
- "**startup**"——不是"启动阶段 / 早期阶段"，而是公司内部专指"**现有业务知识上翻到大仓**"这个工作项。如果把"startup 阶段先做快方案"理解成"开始阶段先做快方案"，会同时错三处：会议要点描述、关键决定措辞、action items 的优先级
- "**蒸馏**"——不是化学/教学比喻，而是"把对话/spec 中的可复用知识自动提炼成 knowledge 文件"
- "**explore**"——不是"探索"动作，而是"**知识图谱遍历检索**"这个具体技术方案
- "**dreaming**"——不是字面英文，而是"**后台异步蒸馏机制**"

判断信号：英文/中文常用词在会议里被反复当名词用、用法不符合字面常识、对应到外部规划文档有具体定义。

处理方式：写之前查"适配小红书内部场景"节的术语字典；字典外的不确定词标"待确认"，会后跟提供者对齐。

### 9. action items 不能从模糊话拼出来

逐字稿里经常出现这样的发言："你明天改完，他周一发出来"——但**谁是"你"、谁是"他"、改什么、发什么都不明确**。

不要硬拼。**三要素 (who / what / when) 任一缺失就不写**，把疑问留到会后单独问。

真实案例：
- 错误做法：把"你们按照上面那个格式再改一改"+"明天改完"+"他周一出来"+"对对对，他的笔记发布"拼成 "桥本 修改文案/格式（按上面格式统一），周一发布笔记"——四句话来自不同时刻，主语指代根本不一致
- 正确做法：跟提供者确认这几句具体指什么，确认不了就不写

宁可纪要 action items 表只有 1 条，也不要为了"看起来完整"凑 5 条。

### 10. 同一天多场连续会议要交叉引用 + 命名能区分

实战中常遇到一天有 3-5 场会议（早会 / 跨人对齐 / 业务方会 / 晚间复盘等），各场互相承接但角色和决策不同。处理时：

- **命名规范**：日期 + 主题外加性质区分词。例：
  - `2026-05-12-BP推进对齐会议纪要.md`（早上小队对齐）
  - `2026-05-12-与久夜对齐新Workspace规范会议纪要.md`（晚 9 点跨人对齐）
  - `2026-05-12-业务BP对焦日会会议纪要.md`(晚 10 点小队复盘）
- **参考链接里互链同期会议**：让读者能跳转到时序前后纪要
- **顶部一句话点明承接关系**：如"延续 X 会议的 Y 决策，本次会议主要校准方案 + 拆解执行"

判断信号：用户提供的逐字稿文件名都是同一日期；议题相互呼应；某场会议提到"昨晚跟 X 对齐过"。

### 11. 战略宣讲 / 1v1 性质会议的特殊处理

对焦会和战略宣讲会的输出形态不同：

| 性质 | 输出重点 | 归属语气 |
|---|---|---|
| 常规对焦会（多人决策） | 关键决定 + 行动项 | 避免归属（用"会议达成共识"） |
| **战略宣讲 / 1v1** | 战略观点 + 核心判断 + 关键金句 | **可以归属**（"X 强调"、"X 判断"），归属是有意义的 |

战略宣讲会特征：单方主导讲解（>70% 时间），少决策多观点，对方多是被动接收者。这种场景下行动项可能很少，但"未来路径"内容很多。

判断信号：参会人 ≤ 3 人；有明显主讲人；议题不是"我们讨论 X"而是"X 给 Y 输入背景 + 判断"。

> 注意：本规则是对"严格禁止 ❌ 用 X 说归属语气"的场景化补充——常规会议仍然避免归属，1v1 战略宣讲场景才允许归属。

### 12. 参会人精确分类（参会 / 旁听 / 未出席被引用）

参会人员列表要严格区分三类身份：

| 身份 | 标注 | 判断依据 |
|---|---|---|
| **参会** | 直接列名 | 在逐字稿里有发言 |
| **旁听** | `X（旁听）` | 用户告知在场但逐字稿无发言 |
| **未出席被引用** | **不列入参会人员**，可在文中说明"会议中引用" | 第三人称被提到（如"X 那边准备得差不多了"），但本人不在场 |

错误案例：把第三人称引用的人列为参会人员 → 让读者误以为他在场表达了观点。

### 13. 参会人推断的证据强度

逐字稿只有"说话人 A/B/C..."标记时，识别真实姓名按证据强度排序：

| 强度 | 证据类型 | 示例 |
|---|---|---|
| ⭐⭐⭐ 强 | 被名字 cue + 立即应答 | A 说"明瑞，你说一下" → G 立刻接话 → G = 明瑞 |
| ⭐⭐⭐ 强 | 自报身份 | "我这边是负责两个 skill" + 上下文匹配某成员的职责定义 |
| ⭐⭐ 中 | 自报具体信息 | "我对接的是 X 仓库" + 已知 BP 表显示 X 仓由 Y 负责 → 可能 = Y |
| ⭐ 弱 | 议题相关性 | 谁讲 explore 多就推断是 explore 负责人 → **易错**（多人合作时） |

仅靠议题相关性不能下结论。不能确定的标"说话人 X / 未识别 / 待用户校准"，由用户校正。

### 14. 分歧未收敛要明确标注，不强行编结论

会议中常有"双方坚持各自方案，没收敛"的情况。处理原则：

- ✅ 明确标"**未收敛 / 待 X 带回讨论 / 后续 Y 会议再定**"
- ❌ 不要为了"看起来有结论"强行编决议
- ✅ 同一议题在多次会议反复出现时，**交代演变**（如"5.12 晚跟 X 讨论时倾向 A 方案；5.13 早会麦督定调采用 B 方案"）

实战例：workspace "灵活组合 vs 业务域固定" 议题在 5.12 晚跟久夜会议未收敛 → 5.13 早 BP 对焦会才定下"业务域固定"——这个演变要交代清楚，否则读者会困惑。

## XMind 大纲生成（可选）

如果用户要求 XMind 梳理，**额外生成**一份 Markdown 大纲文件（命名 `xxx-XMind大纲.md`），规则：

- 用纯 Markdown 列表表达层级，最深 5 层
- 中心主题 = 会议主题
- 一级分支对应纪要的章节
- 末端节点尽量短（关键词级，不要写完整句子）
- 输出后告诉用户三种导入 XMind 的方式：
  1. 复制粘贴（XMind 自动识别 Markdown）
  2. XMind 导入 Markdown 文件
  3. 在小红书 ReDoc 文档中直接粘贴（渲染为可折叠列表，支持二次编辑）

## 文件命名规范

- 会议纪要：`YYYY-MM-DD-{主题}会议纪要.md`
- XMind 大纲：`YYYY-MM-DD-{主题}-XMind大纲.md`
- 默认放在用户当前工作目录，除非用户指定其他位置

## 适配小红书内部场景

### 通用原则

- 如果会议涉及内部系统/服务/中间件，使用准确的内部术语（见下方字典）
- 优先使用中文专有名词（如"Part 平台"而非"Part platform"）
- 人名使用真实姓名/英文名/花名（按现场使用的称呼），不要意译
- 引用内部文档时使用完整的 `docs.xiaohongshu.com/doc/{id}` 链接

### 内部术语字典

字典持续维护，新遇到的术语在用前先查；查不到的标"待确认"会后核对。

**平台 / 工具：**

| 术语 | 含义 |

**概念 / 流程：**

| 术语 | 含义 |

| PE | product engineer 工程师 |
| harness | AI 工程框架（含 init / commit / iter / harvest 等 CLI，参考 Claude Code Harness 设计原则）|
| spec-coding | AI 辅助开发流程（userstory → design → tasks → impl → harvest）|
| 蒸馏 / harvest | 把对话或 spec 中的可复用知识自动提炼成 knowledge 文件 |
| dreaming | 后台异步蒸馏机制 |
| explore | 知识图谱遍历检索 |
| **startup** | **现有业务知识上翻到大仓** —— 注意不是字面"启动阶段"|
| 选仓指南 | 让 AI 判断需求涉及哪几个仓的指南文档 |
| workspace | 多仓项目的根目录 git 仓库，通过 CLI / plugin 把若干业务子仓 + org-knowledge clone 进来统一管理。**不是 IDE/编辑器概念** |
| 自进化 agent | 跑完实验后分析效果、给出优化决策建议、多轮迭代知识库/skill 的 agent |
| 自迭代 agent | 基于 knowledge 调优蒸馏效果、提升不同业务实验场景采纳效果的 agent |
| BP 协作方 / BP 业务方 | BP 协作方 = infra 侧贴身服务的同学；BP 业务方 = 业务团队对接人 |
| org-harness-plugin | 基于 codewiz-cc 的 plugin，提供 spec agent + 知识库上翻 + 探索能力 |
